Tipos de testes

  • Testes de unidade

    Testam uma única unidade do sistema e como elas funcionam individualmente.

  • Testes de integração

    Testa como as unidades trabalham em conjunto, independente de seus funcionamentos individuais.

  • Testes de sistema

    Verificam o funcionamento do sistema como um todo, não se preocupando com o funcionamento das unidades e integrações.

  • Testes de aceitação

    Semelhantes aos testes de sistema porém são mais superficiais. Garantem que o sistema atende aos requisitos mínimos para aceitação.

3 Leis do TDD

  • 1ª lei:

    Nunca escreva código de produção antes de escrever um teste que falha.

  • 2ª lei:

    Só escreva código de teste o suficiente para ter um teste que falha. (Não compilar também é falhar)

  • 3ª lei:

    Só escreva código de produção o suficiente para passar no teste.

Considerações finais

Verdades e recomendações sobre TDD

Clean Code

Seus testes são códigos como qualquer outro e o Clean Code se aplica a eles.

Documentação

Testes de qualidade tendem a se tornar a documentação do seu código.

Testes F.I.R.S.T.

Testes devem ser rápidos; independentes e isolados; repetíveis; auto-validáveis; completos.

Prática

Noções de design não são aprendidas só com teoria, elas são desenvolvidas com estudo constante e muita prática.
